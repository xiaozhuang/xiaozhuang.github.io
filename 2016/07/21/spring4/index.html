IOC

Spring配置的可选方案

  • XML中显示配置
  • Java中显示配置
  • 隐式的bean发现机制和自动装配

建议尽可能地使用自动装配的机制，显示配置越少越好。
推荐使用类型安全且比XML更强大的JavaConfig。
只有需要使用便利地XML命名空间，并且在JavaConfig中没有同样的实现时，才应该使用XML。

自动装配bean

Spring 从两个角度实现自动化装配:

  • 组件扫描（component scanning）：发现应用上下文中所创建的bean。
  • 自动装配（autowiring）：自动满足bean之间的依赖。

Conditional

条件注册

除了自自定义Condition之外，Spring-boot还提供了很多Condition给我们用

SpringBoot源码分析之条件注解的底层实现

@ConditionalOnBean（仅仅在当前上下文中存在某个对象时，才会实例化一个Bean）
@ConditionalOnClass（某个class位于类路径上，才会实例化一个Bean）
@ConditionalOnExpression（当表达式为true的时候，才会实例化一个Bean）
@ConditionalOnMissingBean（仅仅在当前上下文中不存在某个对象时，才会实例化一个Bean）
@ConditionalOnMissingClass（某个class类路径上不存在的时候，才会实例化一个Bean）
@ConditionalOnNotWebApplication（不是web应用）

面向切面的Spring

注解事务

@Transactional 注解

  • value 指定事务管理器
  • propagation 事务传播行为，默认值 REQUIRED
  • isolation 事务隔离度，默认值 DEFAULT
  • timeout 事务的超时时间，默认值为-1，单位是秒
  • readOnly 是否为只读事务，默认值为 false
  • rollbackFor 能够触发事务回滚的异常类型
  • noRollbackFor 指定的异常类型，不回滚事务

事务传播行为

参数 含义
TransactionDefinition.PROPAGATION_REQUIRED 默认值，如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新的事务
TransactionDefinition.PROPAGATION_REQUIRES_NEW 创建一个新的事务，如果当前存在事务，则把当前事务挂起
TransactionDefinition.PROPAGATION_SUPPORTS 如果当前存在事务，则加入该事务；如果当前没有事务，则以非事务的方式继续运行
TransactionDefinition.PROPAGATION_NOT_SUPPORTED 以非事务方式运行，如果当前存在事务，则把当前事务挂起
TransactionDefinition.PROPAGATION_NEVER 以非事务方式运行，如果当前存在事务，则抛出异常
TransactionDefinition.PROPAGATION_MANDATORY 如果当前存在事务，则加入该事务；如果当前没有事务，则抛出异常
TransactionDefinition.PROPAGATION_NESTED 如果当前存在事务，则创建一个事务作为当前事务的嵌套事务来运行；如果当前没有事务，则该取值等价于REQUIRED

事务隔离级别

参数 含义
TransactionDefinition.ISOLATION_DEFAULT 默认值，默认隔离级别
TransactionDefinition.ISOLATION_READ_UNCOMMITTED 未提交读（Read uncommitted）
TransactionDefinition.ISOLATION_READ_COMMITTED 已提交读（Read committed）
TransactionDefinition.ISOLATION_REPEATABLE_READ 可重复读（Repeatable read）
TransactionDefinition.ISOLATION_SERIALIZABLE 可串行化（Serializable）

事务的只读属性

如果确定只对事务性资源进行只读操作，那么我们可以将事务标志为只读的，以提高事务处理的性能

取决于JDBC实现，并不是所有都支持的；MySQL支持，Oracle不支持

事务的回滚规则

默认情况下，如果在事务中抛出了未检查异常（继承自 RuntimeException 的异常）或者 Error，则 Spring 将回滚事务；除此之外，Spring 不会回滚事务。

注解方式的事务使用注意事项

@Transactional 只能应用到 public 方法才有效
若不是 public，就不会获取@Transactional 的属性配置信息，最终会造成不会用 TransactionInterceptor 来拦截该目标方法进行事务管理

不加@Transactional

不管写不写transaction，事务是肯定有的。
只是默认单条sql一个事务，
如果我们在Service层的方法上加Transaction注解，那么事务的范围就会扩大，包含在service方法执行的所有sql
