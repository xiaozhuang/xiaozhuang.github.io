Java 设计模式

面向对象设计原则

单一职责原则

类的职责要单一，不能将大多的职责放在一个类中。

开闭原则

软件实体对扩展是开放的，但对修改是关闭的。
即在不修改软件实体的基础上去扩展其功能。

里氏替换原则

在软件系统中，一个可以接受基类对象的地方，必然可以接受一个子类对象。

依赖倒转原则

要针对抽象层编程，而不要针对具体类编程。

接口隔离原则

使用多个专门的接口来取代一个统一的接口。

合成复用原则

在系统中应该尽量多使用组合和聚合关系，尽量少使用甚至不适用继承关系。

迪米特法则

一个软件实体对其他实体的引用越少越好，
或者说如果两个类不必彼此直接通信，
那么这两个类就不应当发生直接的相互作用，
而是通过引入一个第三者发生间接交互。

策略模式（Strategy Pattern）

定义了算法族，分别封装起来，让他们之间可以相互替换，
此模式让算法的变化独立于使用算法的客户。

观察则模式

定义了对象之间的一对多依赖，当对象改变状态时，
依赖它的对象会收到通知，并自动更新。

观察者模式让主题和观察者之间松耦合。

使用场景（购票为核心业务）：

  • 购票后记录文本日志
  • 购票后记录数据库日志
  • 购票后发送短信
  • 购票送抵扣卷、兑换卷、积分
  • 其他各类活动等

HeadFirst示例：气象站布告板

装饰者模式

动态地将功能附加到对象上。

使用组合和委托运行时动态扩展行为；

使用场景：
Java I/O、Java线程安全结合包装器、RequestWrapper

HeadFirst示例：星巴兹咖啡订单系统，包装添加不同调料的咖啡

工厂模式

简单工厂模式

简单工厂模式由一个工厂决定创建出哪一种产品类的实例。

工厂方法模式

定义一个创建对象的接口，但由子类决定要实例化的类是哪一个。
工厂方法模式让类把实例化推迟到子类。

工厂类不再负责所有产品的创建，而是将具体创建的工作交给子类。

加入新产品，只需向系统中加入这个产品及其所对应的工厂类。

HeadFirst示例：不同地区披萨店

抽象工厂模式

抽象工厂模式可以向客户端提供一个接口，使客户端在不指定产品的具体类型情况下，
创建多个产品族中的产品对象。

提供一个接口，用于创建相关或依赖对象的家族，并不需要明确指定具体类。

抽象工厂模式可以向客户端提供一个接口，使客户端在不指定产品的具体类型情况下，
创建多个产品族中的产品对象。

使用场景：电脑配件工厂（pc、mac）
HeadFirst示例：不同地区披萨原料工厂

适配器模式与外观模式

适配器模式

将一个类的接口，转换成客户期望的另一个接口。
适配器让原本接口不兼容的类可以合作无间。

外观模式

提供一个统一的接口，用来访问子系统中的一群接口。
外观定义了一个高层接口，让子系统更容易使用。

图片来源Facade Design Pattern

外观不只简化了接口，也将客户从组件的子系统中解耦。

模板方法模式

在一个方法中定义一个算法的骨架，而将一些步骤延迟到子类中。
模板方法使得子类可以在不改变算法结构的情况下，重新定义算法中的某些步骤。

模板方法的抽象类可以定义具体方法、抽象方法和钩子。

钩子可以不做任何事情或只做默认的事情。

钩子可以让子类实现算法中可选的部分，
或让子类对模板中某些即将发生（或刚刚发生）的步骤作出反应。

使用场景：SpringMVC HandlerMethodAdapter（子类RequestMappingHandlerAdapter）
Spring MVC ViewResolver（子类）

状态模式

通过改变对象内部状态帮助对象控制行为。

Context将行为委托给当前状态

状态模式和策略模式类图一样，区别在于“意图”

策略模式通常用行为类或算法来配置Content，
可作为除继承外的弹性替代方案。

状态模式允许Context随状态的改变而改变行为，
可作为Context中放置很多条件判断的替代方案。

使用抽象类实现状态模式，抽象类中容易添加新方法，
不需要打破具体状态实现。

状态转换放置在状态类中的缺点：状态类之间产生依赖。
通过使用Context上的getter方法把依赖减到最小。

使用场景：酒店房间，未预定状态、已预定状态

HeadFirst示例：糖果机，等待投币、已投币、正在出售、已售罄

代理模式

为另一个对象提供一个替身或占位符以控制对这个对象的访问。

应用场景：

  • 远程代理
  • 虚拟代理
  • 保护代理

虚拟代理：作为创建开销大的对象的代表。如：ImageProxy，图片加载完成前输出提示信息，加载完成刷新显示

静态代理：代理类是在编译时就实现好的。代理类是一个实际的 class 文件。
动态代理：代理类是在运行时生成的。在运行时动态生成的类字节码，并加载到JVM中。

1.使用动态代理，动态为接口生成实现类(仅写接口不写实现)

2.动态代理保护对象访问

JDK中所要进行动态代理的类必须要实现一个接口，也就是说只能对该类所实现接口中定义的方法进行代理，这在实际编程中具有一定的局限性，而且使用反射的效率也并不是很高。

CGLIB是一个功能强大，高性能的代码生成包。它为没有实现接口的类提供代理，为JDK的动态代理提供了很好的补充。

CGLib不能对声明为final的方法进行代理

CGLIB原理：动态生成一个要代理类的子类，子类重写要代理的类的所有不是final的方法。
在子类中采用方法拦截的技术拦截所有父类方法的调用，顺势织入横切逻辑。
它比使用java反射的JDK动态代理要快。

CGLIB底层：使用字节码处理框架ASM，来转换字节码并生成新的类。
