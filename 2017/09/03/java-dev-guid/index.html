Java 开发规范

发表于 2017-09-03   |   分类于 Note

《阿里巴巴Java开发手册》阅读笔记

编程规约

命名规范

  • 抽象类命名使用Abstract或Base开头；异常类命名使用Exception结尾；测试类以被测试类名开始，以Test结尾
  • 中括号是数组类型的一部分，数组定义如下：String[] args
  • POJO类中布尔类型的变量，都不要加is
  • 包名统一使用小写，点分隔符之间有且仅有一个自然语义的英语单词，包名统一使用单数形式，类名可以使用复数形式
  • 枚举类名建议带上Enum后缀，枚举成员名称全大写

Service/DAO层方法命名规约
1)获取单个对象get前缀
2)获取多个对象list前缀
3)统计值方法用count前缀
4)插入用save或insert前缀
5)删除用remove或delete前缀
6)修改的方法用update做前缀

领域模型命名规约
1)数据对象，xxxDO
2)数据传输对象，xxxDTO
3)展示对象：xxxVO
4)POJO是DO/DTO/BO/VO的统称

常量定义

  • 不允许任何魔法值
  • long或者Long初始赋值时，必须使用大写的L
  • 不要使用一个常量类维护所有常量，按常量功能进行分类。缓存CacheConsts、系统配置ConfigConsts
  • 常量声明放在constant目录下

代码格式

  • 任何二目、三目运算符的左右两边都必须加一个空格
  • 缩进采用4个空格，禁止使用tab字符
  • 单行字符数限制不超过120个
  • 文件编码格式UT-8；IDE中文件的换行符使用Unix格式，不要使用windows格式

OOP规约

  • 避免使用对象访问静态变量或静态方法
  • 所有的覆盖方法，必须加@Override注解
  • 不能使用过时的类或方法
  • Object的equals方法容易抛空指针异常，推荐使用java.util.Objects.equals(Object, Object)（JDK7引入的工具类）
  • 所有的相同类型的包装类对象之间的值的比较，全部使用equals方法
  • 所有的POJO类属性必须使用包装数据类型
  • RPC方法的返回值和参数必须使用包装数据类型
  • 所有局部变量使用基本数据类型

  • 定义DO/DTO/VO等POJO类时，不要设定任何属性默认值

  • 构造方法里面禁止加入任何业务逻辑，初始化逻辑，放在init方法中

  • POJO类必须写toString方法

  • 循环体内，字符串的连接方式，使用StringBuilder的append方法

  • 集合转数组使用集合的toArray(T[] array)

  • 数组转集合使用Arrays.asList()

  • 不要在foreach循环里进行元素的remove/add操作。remove元素使用Iterator方式

  • 集合初始化时，指定集合初始化大小

  • 使用entrySet遍历Map类集合KV，而不是keySet方式；JDK8使用Map.foreach方法

并发处理

  • 线程资源必须通过线程池提供

  • 线程不允许使用Executors去创建，而是通过ThreadPoolExecutor的方式

  • SimpleDateFormat 是线程不安全的类，定义static必须加锁，或使用DateUtils工具类

JDK8使用Instant代替Date，LocalDateTime代替Calendar，
DateTimeFormatter代替Simpledateformatter，官方解释：simple beautiful strong immutable thread-safe

  • 并发修改同一记录时，避免更新丢失，需要加锁。缓存加锁或数据库乐观锁。

访问冲突概率小于20%，推荐使用乐观锁，否则使用悲观锁。乐观锁重试次数不小于3次。

  • 避免Random实例被多线程使用，虽然共享实例线程安全，但因竞争同一seed导致性能下降。(JDK7之后，使用API ThreadLocalRandom)
  • volatile解决多线程内存不可见问题。对于一写多读，可解决变量同步问题。

JDK8推荐LongAdder对象，比AtomicLong性能更好(较少乐观锁重试次数)

控制语句

  • 复杂逻辑判断的结果赋值给一个有意义的布尔变量，以提高可读性。

  • 需要参数校验场景：执行开销大的方法；对外开放接口；敏感权限接口。

  • 不需要参数校验场景：底层调用频度较高的方法；private方法；

注释规约

  • 类、类属性、类方法注释必须使用Javadoc规范
  • 所有抽象方法必须用Javadoc注释，除返回值、参数、异常说明外，还必须指出方法功能
  • 所有类都必须添加创建者和创建日期
  • 方法内部单行注释，在被注释语句上方另起一行
  • 枚举类型字段必须注释
  • 特殊注释标记：待办事宜（TODO）:（ 标记人，标记时间，[预计处理时间]）；错误，不能工作（FIXME）:（标记人，标记时间，[预计处理时间]）

其他

  • 使用正则表达式时，利用预编译功能。不要在方法体内定义：Pattern pattern = Pattern.compile(规则);
  • 注意 Math.random() 这个方法返回是double类型，注意取值的范围 0≤x<1（能够取到零值）
  • 获取当前毫秒数System.currentTimeMillis(); 而不是new Date().getTime();在JDK8中，推荐使用Instant类
  • 对于“明确停止使用的代码和配置”，坚决从程序中清理出去，避免造成过多垃圾。

异常日志

  • 最外层的业务使用者，必须处理异常
  • 不能在finally块中使用return，finally块中的return返回后方法结束执行，不会再执行try块中的return语句。
  • 防止NPE：远程调用返回对象时，一律要求进行空指针判断；使用JDK8的Optional类来防止NPE问题
  • 在代码中使用“抛异常”还是“返回错误码”：公司外的http/api开放接口必须使用“错误码”；而应用内部推荐异常抛出；跨应用间RPC调用优先考虑使用Result方式，封装isSuccess()方法、“错误码”、“错误简短信息”。

日志规约

  • 不可直接使用日志系统中的API，应该依赖日志框架SLF4J中的API
  • 日志文件推荐至少保存15天

  • 对trace/debug/info级别的日志输出，必须使用条件输出格式或占位符的方式

  • 异常信息应该包括两类信息：案发现场信息和异常堆栈信息。

  • 可以使用warn日志级别来记录用户输入参数错误的情况
  • error级别只记录系统逻辑出错、异常等重要的错误信息

MySQL数据库

建表规约

  • 表达是否的字段，必须使用is_xxx的方式命名，类型unsigned tinyint（1表示是，0表示否）；任何字段如果为非负数，必须是unsigned。
  • 表名、字段名必须使用小写字母和数字，禁止数字开头、禁止两个下划线中间只出现数字
  • 表名不用复数名词
  • 主键索引pk_字段名、唯一索引uk_字段名、普通索引idx_字段名
  • 小数类型为decimal，禁止使用float和double
  • 存储的字段长度几乎相等，使用char定长字符串类型
  • varchar长度不要超过5000
  • 单表数据超过500W或单表容量超过2GB，才推荐分库分表
类型 表示范围
unsigned tinyint 无符号值：0到255
unsigned smallint 无符号值：0到65535
unsigned int 无符号值：0到约42.9亿
unsigned bigint 无符号值：0到约10的19次方

TINYINT[(M)] [UNSIGNED] [ZEROFILL]
A very small integer. The signed range is -128 to 127. The unsigned range is 0 to 255.
M indicates the maximum display width for integer types.

无符号，TINYINT 高位按M补0

索引规约

  • 业务上具有唯一特性的字段，即使是多个字段的组合，也必须建成唯一索引。提高查找速度。
  • 超过3个表，禁止join；即使双表join也要注意表索引、SQL性能

  • varchar字段建立索引，必须制定索引长度

  • SQL性能优化的目标：至少要达到 range 级别，要求是ref级别，如果可以是consts最好。

1）consts 单表中最多只有一个匹配行（主键或者唯一索引），在优化阶段即可读取到数据。
2）ref 指的是使用普通的索引（normal index）。
3）range 对索引进行范围检索。

  • 防止因字段类型不同造成的隐式转换，导致索引失效

SQL语句

  • 不要使用count(列明)或count(常量)来代替count(*)
  • 当某一列的值全是NULL时，count(col)的返回结果为0，但sum(col)的返回结果为NULL，因此使用sum()时需注意NPE问题
  • in操作尽量避免，in后边的集合元素数量，控制在1000个之内

ORM映射

  • 不要用resultClass当返回参数，即使所有类属性名与数据库字段一一对应，也需要定义

工程结构

应用分层

  • 开放接口层：RPC接口、HTTP接口、进行网关安全控制、流量控制
  • 终端显示层:各个端的模板渲染并执行显示的层
  • Web层：对访问控制进行转发，各类基本参数校验，或不复用的业务简单处理

  • Service层：具体业务逻辑服务层

  • Manager层：通用业务处理层，第三方平台封装层、Service层通用能力下沉（缓存、中间件通用处理）、DAO层交互

  • DAO层：数据访问层

在DAO层，不需要打印日志，因为日志在Manager/Service层一定需要捕获并打到日志文件中去

分层领域模型

  • DO（Data Object）：与数据库表结构一一对应
  • DTO（Data Transfer Object）：数据传输对象，Service和Manager向外传输的对象
  • BO（Business Object）：业务对象。可以由Service层输出的封装业务逻辑的对象。
  • Query：数据查询对象，各层接收上层的查询请求。注：超过2个参数的查询封装，禁止使用Map类来传输。
  • VO（View Object）：显示层对象，通常是Web向模板渲染引擎层传输的对象。
